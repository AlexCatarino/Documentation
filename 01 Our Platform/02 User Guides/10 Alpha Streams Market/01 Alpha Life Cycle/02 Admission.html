Algorithms can only be listed in the Alpha Streams Market if their backtest passes the following requirements:

  • Length: The backtests must be 5-7 years in length.
  • End date: The backtest must have an end date less than 7 days prior to the date of submission.
  • Reality modeling: The backtest must use the AlphaStreamsBrokerageModel and cannot include any additional reality modeling.
  • Profitability: The backtest must be profitable.
  • Data sources: The backtest must only use datasets from our Dataset Market because we can't guarantee the integrity of any external data sources. Refer to the License tab on a dataset listing to see if the dataset is eligible for use in Alpha Streams.
  • PSR: The backtest must have a Probabilistic Sharpe Ratio (PSR) >= 80%.
  • Drawdown: The backtest must have a maximum drawdown <= 20%. If the maximum drawdown >= 10%, then the maximum drawdown duration must be <= 6 months.
  • Capacity: The backtest must have a capacity >= $250,000.
  • Correlation: The backtest must have a return correlation <= 85% with other Alphas that have been accepted by the same Author.
  • First trade: The backtest must place a trade on deployment.
  • Asset classes: The backtest must not trade Futures or Options.
  • Order count: The backtest must place more than 10 orders per month.

Alphas do not need to perform well in all market environments to be accepted into the Alpha Streams Market. Licensees can adjust their allocation to Alphas as the market environment changes.

If an Alpha passes the criteria above, it is reviewed by our team. We aim to not impose any value bias during the review process, but we need to ensure the Alphas are mechanically sound. When reviewing Alphas, we consider the following points:

Stateless & Resilient

Mechanisms should be set up to restore the internal state of the Alpha in the event of reboots. Alphas will run for a very long period of time. Over the course of months or years, the server will need patches and occasional restarting. The Alpha should be smart enough to automatically recover in the event of restarts using the History and WarmUp methods.

Grounded in Reality

Alphas should have clear strategic reasoning underpinning the strategy. A single reason-based foundation for the algorithm is necessary to understand the Alpha behavior when things go wrong and it helps improve your Alpha application for funds. Obscure correlations or overfit strategies tend to perform poorly out-of-sample.

Transparent Sourcing

Authors need a transparent employment history and Alphas must not infringe on other intellectual property (IP). We like to know the employment history of Authors on the platform to ensure their Alphas are not infringing on any IP, including current and past employers. Public or shared algorithms may be used as a foundation and extended. If you have licensed the IP from a third party, you must present consent for using the IP.

Edge Case Handling

Alphas need to be able to handle edge cases such as dividends, splits, and delistings. Handling edge cases adds another layer of resiliency to Alphas and keeps them running when data changes occur. Authors can handle edge cases with simple control logic, checking data.ContainsKey(symbol), explicitly removing certain affected securities from the universe of traded securities, or other methods.

Insights

Insights are predictions about the price movement of specific securities. Insights are required in all Alpha Streams submissions. If the algorithm follows the classic design, it must call the EmitInsights method before each order is submitted. Insights provide Licensees with information about the predictive power of Alphas and give the Licensees insight into why the orders are being placed. If the algorithm follows the framework design, the Update method of the Alpha model must return a list of Insights.

Daily Data

We aggregate our daily data from ticks and then inject the daily bars into the algorithms at 00:00 UTC. For instance, TradeBars with a daily resolution that span across the day 2019-10-22 are injected into algorithms at 2019-10-23 00:00 UTC. If an algorithm uses daily data, operations performed intraday will be done using stale data, leading to unexpected results, impossible order fills, and unrealistic performance. To perform operations intraday while maintaining realistic performance, use minute or hour resolution and place a Scheduled Event.

Open-Source IP

We provide the community with lots of example algorithms. The goal of the examples is to demonstrate how to use the API correctly, incorporate new data sources into existing algorithms, implement best practices for Alpha Streams, and more. However, we cannot accept any copies or near-copies of our demonstration algorithms into Alpha Streams. We love to see that our work inspired someone, but each submission must provide sufficient originality that the work can indeed be called the Author's own.

Overfitting

Overfitting will doom algorithms in live trading. To prevent overfitting and boost the quality of submissions, we can't accept any Alphas that obviously overfit to data. Overfitting can manifest itself in countless ways, but we most commonly see the following:

  • Setting indicator parameters that work for certain hand-picked assets but perhaps not for any others.
  • Using thresholds for indicator values that are hard-coded and have no fundamental theory behind their value.
  • Look-ahead bias, such as hard-coding specific dates to perform specific actions. Hard-coding specific dates can only be done if there is prior knowledge of an event. Look-ahead bias might boost the backtest performance, but it does not mean the Alpha is fundamentally valuable and can sustain performance during future events.
  • Selection bias, such as picking stocks known ahead of time to perform exceptionally well during specific periods.

Infrequent Insight Generation or Trading

Algorithms don't need to emit Insights, trade daily, or trade intraday, but the maximum holding period for funds is usually a matter of days or weeks. We likely won't accept Alphas that rebalance less frequently than monthly. Alphas need to place at least 10 trades per month for the majority of the backtest.

IP Infringement

Authors can only submit Alphas to which they own the IP. As Authors submit Alphas, they must include information on themselves and meta-information about the algorithm. The information that's collected is displayed on the Alpha listing page for investors to review. We collect the Author's biography, work experience, and education. We also collect the following information on each Alpha:

  • Name
  • Description
  • Reserve price
  • Image
  • Type of universe selection
  • Type of alpha generation
  • Type of portfolio construction
  • Type of risk control
  • Type of execution model(s)

If an algorithm is accepted to the Alpha Streams Market, it is immediately listed in the market. If an algorithm does not meet the admission requirements above, the algorithm is rejected and our Alpha Streams team provides comments on why the algorithm was rejected. When an algorithm is rejected, the Author can update the algorithm and then submit the updated version for review.
